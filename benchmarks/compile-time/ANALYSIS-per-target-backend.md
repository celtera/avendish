# Compile-time: ranked cost per-target and per-backend (real examples)

Companion to `ANALYSIS-ftime-trace.md`. That doc drilled into *where* the time
goes on one object; this one answers the broader question across **all examples
and as many backends as run locally**: what is the ranked cost per target, and
per backend?

## Method
- **Objects**: every SDK-free `dump`-backend translation unit (one per registered
  example, generated by `.github/scripts/gen_dump_tus.py`). 42 of 46 compile with
  no external device SDK; the other 4 need `chowdsp_wdf` / `rnd/random` / a Tutorial
  template placeholder and are skipped on every config.
- **Backends measured locally** (4): `clang20-pfr`, `clang22-pfr`,
  `clang22-p1061` (structured-binding flattener), `gcc14-pfr`. clang ≥ 21 / gcc ≥ 16
  (the P1061-default toolchains) are exercised by the `Backend matrix` CI workflow
  and the docker runs in `ANALYSIS-ftime-trace.md`.
- **Flags**: `-std=c++23 -O0 -c`, best-of-3. `-O0` on purpose: it isolates the C++
  **frontend / template-instantiation** cost, which is the only thing the
  introspection backend influences. `-O2` is codegen-bound and backend-blind (every
  config lands within noise of every other), so it cannot answer "which backend".
- Raw data: `results/examples_o0_local.csv`.

## Headline: the cost is a fixed per-TU floor, not the backend

| component (clang-22, -O0) | time | share of a typical example |
|---|---:|---:|
| empty TU | 0.01 s | — |
| `#include <nlohmann/json.hpp>` | **1.37 s** | 62 % |
| + `<fmt/format.h>` | 1.57 s | 71 % |
| + avnd dump headers (no object) | **1.72 s** | 78 % |
| mean *real* example | ~2.2 s | 100 % |

So **~1.7 s of every ~2.2 s example is fixed harness header overhead** — and
`nlohmann/json` alone is 62 % of it. The avendish introspection layer is ~0.15 s
of headers plus ~0.5 s marginal per object, and is **backend-invariant**: on the
same compiler, the structured/P1061 flattener vs boost.pfr is **0.98× mean** across
42 examples (noise). The floor is a property of the `dump` binding (fmt + json), not
of avendish or of the reflection work.

## Per-backend ranking (42 examples built by all configs, -O0, best-of-3)

| config | sum (s) | mean (s) | median | max |
|---|---:|---:|---:|---:|
| gcc14-pfr | 90.6 | 2.156 | 2.037 | 6.55 |
| clang20-pfr | 93.4 | 2.224 | 2.121 | 5.96 |
| clang22-p1061 | 97.2 | 2.314 | 2.213 | 5.38 |
| clang22-pfr | 99.0 | 2.356 | 2.248 | 5.88 |

The spread between the fastest and slowest config is **9 %**, and it tracks the
*compiler's* baseline parse speed (gcc14 < clang20 < clang22), not the backend. On
one compiler, clang22-p1061 beats clang22-pfr (97.2 vs 99.0 s total, 0.98×) — the
P1061 flattener is marginally cheaper, never costlier. On typical objects the win is
sub-noise; it only becomes visible on the one outlier below.

## Per-target ranking (mean across the 4 configs, -O0)

One object dominates; the rest are a flat ~2.0–2.6 s band (the floor + a small,
port-count-proportional tail).

| rank | example | mean (s) |
|---:|---|---:|
| 1 | **examples__AllPortsTypes** | **5.94** |
| 2 | helpers__CompleteMessageExample | 2.57 |
| 3 | uo__NoiseBuffer | 2.56 |
| 4 | helpers__Ui | 2.40 |
| 5 | vb_ports__fourses_tilde | 2.31 |
| … | (35 more, 1.98–2.27 s) | ~2.1 |
| 42 | examples__Construct | 1.98 |

`AllPortsTypes` is the stress object: one struct of ~50 ports spanning every port
type, including `std::unordered_map` ports. It is the only target where the per-TU
floor is a minority of the cost — and the only one where the backend choice is
measurable: **5.38 s (p1061) vs 5.88 s (pfr) — 0.91×.**

## Where the AllPortsTypes delta comes from (`-ftime-trace`)

The structured backend's saving is one entry, cleanly isolated:

| entry | boost.pfr | P1061 |
|---|---:|---:|
| `CheckConstraintSatisfaction <aggregates.pfr.hpp:97>` (boost.pfr field-counting SFINAE) | **588 ms** | **— (absent)** |
| `OptModule` (-O0 codegen) | 594 ms | 594 ms |
| `ParseDeclarationOrFunction AllPortsTypes.hpp:51` (the ~50-field struct) | 180 ms | 172 ms |
| binding `vformat` / `print_*` (dump fmt/json) | ~120 ms | ~115 ms |
| **Frontend total** | **6062 ms** | **5535 ms** |

The whole 0.91× comes from deleting boost.pfr's recursive field-count SFINAE
(`fields_count` / `detect_fields_count` behind `aggregates.pfr.hpp:97`). Everything
else — codegen, parsing the struct, the dump binding's formatting — is identical.

## Conclusions (ranked by leverage)

1. **For real-world objects, the backend is not the bottleneck — the `dump`
   harness's `nlohmann/json` include is** (1.37 s fixed per TU). If `dump` compile
   time ever matters, precompile/extern-template the json/fmt layer; that dwarfs any
   introspection change. (Not an avendish-core issue.)
2. **The P1061/structured backend is a strict, if small, win on normal objects**
   (0.98× mean) and a real one on wide structs (0.91× on AllPortsTypes), because it
   removes boost.pfr's field-counting SFINAE. It is never slower. This is the same
   mechanism as the large synthetic wins in `ANALYSIS-ftime-trace.md` (flat-256:
   82 s → 1.8 s) — those are just the asymptote of the AllPortsTypes effect.
3. **Backend choice is irrelevant to per-target *ranking*.** All four configs
   produce the same ordering; the expensive objects are expensive because they have
   many ports/instantiations, not because of the backend.

Net: the reflection/structured migration removed the one super-linear cost
(`tuple_cat` + consteval re-evaluation, #110) and shaves a further ~9 % of frontend
time off wide structs by dropping boost.pfr's SFINAE field-counting — with zero
regression and a small uniform win everywhere else.

# Avendish test-harness plan

Goal: a battery of "unit-test" plugin objects that exercise every corner of every
backend (every I/O type × feature × supported format), plus programmatic tester
patches per backend, plus fuzz coverage.

## Pieces

1. **Feature catalog** — `AVND_FEATURE_CATALOG.md` (generated by analysis): the
   exhaustive list of API features → drives which test objects we need.
2. **Test objects** — small avnd/halp structs under `examples/Tests/` (proposed),
   one per feature cluster (controls, every texture format, audio poly, midi,
   tensor, buffer, geometry, messages, dict, metadata, scheduling…). Each is built
   for every backend via the existing `avnd_make_*` machinery.
3. **Tester-patch generators** — one tool that reads each object's **dump JSON**
   and emits a tester patch per backend. The dump JSON is the universal contract.
4. **Fuzz** — automatic: `cmake/avendish.fuzz.cmake` + `binding/fuzz/fuzzer.hpp`
   build a libFuzzer target per object from `prototype.cpp.in`. Building the test
   objects with the fuzz backend enabled gives per-object fuzzers for free.

## The dump JSON contract (the key enabler)

Every object already emits `dump/<c_name>.json`:
```
metadatas: { name, c_name, uuid, category, description }
features:  { inputs: value|..., outputs: ... }
inputs[]/outputs[]: { name, type: texture|parameter|audio|midi|message|...,
   parameter: { value_type: float|int|bool|enum|string|unknown,
                control, range:{min,max,init}, choices[], widget:
                slider|knob|xy|choices|toggle|... } }
```
A single generator consumes this and produces, per object, a tester patch that
puts a driver on every input (number/message/slider) and a sink on every output
(print/scope/number). No backend-specific introspection needed.

## Per-backend tester-patch generation

- **Max/MSP** — emit `.maxpat` (JSON: `patcher.boxes[]` with
  `maxclass:"newobj", text:"<c_name> args"`, number/message/`print`/`jit.pwindow`
  boxes per port; `patcher.lines[]` patchlines `[srcid,outlet]->[dstid,inlet]`).
  Format reference: `test_generator.maxpat`, `docs/max-examples.maxpat`,
  `docs/max-jitter-examples.maxpat`.
- **Pure Data** — emit `.pd` netlist (`#N canvas …; #X obj x y <name> args; #X
  floatatom/msg/bng …; #X connect src outlet dst inlet;`). Reference:
  `docs/pd-examples.pd`, `examples/Complete/CompleteMessageExample.pd`.
- **TouchDesigner** — emit a `.py` that drives TD's Python API (`op('/').create(...)`,
  parameter pars, a Constant/Slider per input, a Null + viewer per output, save .toe).
- **Godot** — emit a `.tscn` scene (or a GDScript) instantiating the GDExtension
  node and a driver script setting properties / feeding a texture.
- **WASM / standalone** — the standalone backend already generates a web UI per
  object; collect those pages into a test index (the "patch" is the generated HTML).
- **score** — emit a `.js` using the JS scripting API `EditJsContext`
  (`src/plugins/score-plugin-js/JS/Qml/EditContext.hpp`): `rootInterval()` →
  `createProcess(interval, "<name>", "")` → `inlet/outlet/port` + `setValue` +
  `automate` + `createCable` + `play()` + `serializeAsJson()`. Names come from
  `availableProcesses()`. (See ossia/score-docs for the JS scripting reference.)
- **Python** — emit a `.py` that `import`s each module and calls it with sample
  inputs derived from the dump ranges.

### Run fuzzing (per-object libFuzzer + ASan; automatic targets)
Reconfigure `-DAVND_ENABLE_FUZZ=ON -DBUILD_TESTING=ON`, build `<Target>_fuzz`, run.
On Windows clang-cl the working recipe is (bypasses this tree's msys64 Boost):
`clang-cl -fsanitize=fuzzer,address -D_DISABLE_STRING_ANNOTATION=1 -D_DISABLE_VECTOR_ANNOTATION=1 -clang:-std=c++2c -Zc:__cplusplus /bigobj -I<avnd/include> -I<magic_enum> -I<qlibs_reflect> -I<clean boost> <Class>_fuzz.cpp -Fe:fz.exe`
then `PATH=D:/llvm/lib/clang/<v>/lib/windows;%PATH%` and `fz.exe -max_total_time=N`.
On Linux/CI just `ctest -L fuzz`. TestFloatSlider: 1.6M runs/11s, ASan, no crash.

## Automated backend testing
`tooling/run_backend_tests.py --dumps <dir> --out <dir> [--pd PATH --max PATH
--godot PATH --score PATH --python-modules DIR --td-report FILE]`
- Generates all backends (`gen_tester_patches.py --backend all`): per-object
  patches + `td-runner/td_run_all.py` + `python-runner/py_run_all.py` + wasm index.
- Max/Pd patches self-drive via `loadbang` (loading = instantiate + one cook =
  the crash/load-error smoke test), so a headless launch that survives = pass.
- Runs each host headless if its binary is given; writes `report.json` + summary.
- TD: run `td_run_all.py` inside TouchDesigner (creates/cooks every Custom OP,
  records `errors()`), then pass its `td_test_report.json` via `--td-report`.

### Verified: headless Pd (real run)
`run_backend_tests.py --dumps <d> --out <o> --pd <pd.exe> --pd-externals <dir-with-c_name.dll>`
8/8 test objects loaded+instantiated in headless Pd 0.55 (float/int slider, toggle,
enum, value, messages, midi, lineedit), no crashes. Recipe: patches self-quit
(`--autoquit`) and run `pd -nogui -mmio -noprefs -stderr -path <dir> -open <patch>`
(-mmio/-noprefs avoid a JACK-configured pd). Pass = `<c_name>.dll … succeeded`
and no `couldn't create`. Build a pd external: clang-cl /LD on `<Class>_pd.cpp`
+ `-ID:/apps/pd-0.55-1/src` + link `pd.lib`, output `<c_name>.dll`.

### TouchDesigner (real run)
Install the built `<obj>_TOP_td.dll` into `Documents/Derivative/TouchDesigner/Plugins`.
Registration smoke (verified): launching TD with all 21 cv plugins present ran 30s
with no crash — the FillTOPPluginInfo path is sound. Full per-op cook: run the

errors -> td_test_report.json), then `run_backend_tests.py --td-report <file>`.
(The .toe text format is binary, so the harness DAT is a one-time GUI step.)

### Max (real run)
`run_backend_tests.py --max <Max.exe> --max-externals <dir-with-.mxe64>`: patches
self-quit (`--autoquit`, loadbang -> del -> `; max quit`) and are co-located with
the .mxe64 (Max searches the patch's folder). A clean exit (code 0) = the external
instantiated and Max quit without crashing (verified once for cv_camshift, the
enum object). Max is single-instance + GUI with no headless mode, so batch runs
are flaky (recovery dialogs); run patches individually for a reliable signal.
